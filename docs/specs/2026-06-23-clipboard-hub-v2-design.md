# Clipboard Hub V2 — 设计文档

> 创建日期：2026-06-23
> 当前修订：2026-06-24

> 基线：现有 PySide6 桌面实现

## 一、目标

Clipboard Hub V2 是一个 Windows 本地剪贴板历史工具。当前版本的目标口径如下：

1. 历史记录在程序退出后仍然保留。
2. 历史列表优先服务文字阅读和纯文本重新复制。
3. 剪贴板里的文件列表显示为本地文件路径文本。
4. 图片剪贴板不再作为图片保存、预览或写回。
5. 保留明确的复制成功/失败视觉反馈，同时降低列表噪音。
6. 面板 UI 采用冷白微蓝的 iOS 玻璃风格紧凑列表，便于快速扫描。

数据仅保存在本机，不加密，不上传，不同步。

## 二、当前架构

```text
Windows Clipboard
       |
       v
ClipboardWatcher
  - 轮询剪贴板
  - 读取 text/html/file-url
  - 获取来源应用
  - 生成内容指纹
       |
       v
ClipboardStore
  - 内存缓存
  - 连续重复去重
  - 1000 条 / 200 MiB 保留策略
  - SQLiteRepository
       |
       v
HistoryPanel
  - 搜索
  - 紧凑文本列表
  - 纯文本写回
  - 成功/失败反馈
  - 鼠标移出后自动收回
```

模块职责：

- `ClipboardWatcher`：轮询剪贴板、解码 MIME 数据、采集来源应用、避免自复制回流。
- `ClipboardStore`：负责增删查、连续去重、保留策略和 UI 变更信号。
- `SQLiteRepository`：负责 SQLite 初始化、读写、删除、总量统计和损坏恢复。
- `TabBar`：负责悬浮触发、拖动与当前显示器吸附。
- `HistoryPanel`：负责搜索、历史列表、删除确认、纯文本复制反馈和收回逻辑。

## 三、持久化设计

### 3.1 存储方案

使用 SQLite，不使用 JSON。当前目标行为不保存新的图片载荷；数据库仍保留图片相关字段，用于兼容旧数据和既有迁移成本。

默认数据库文件名是 `clipboard_hub.db`。路径解析顺序以代码为准：

1. `QStandardPaths.AppDataLocation`
2. `QStandardPaths.GenericDataLocation`
3. `LOCALAPPDATA`
4. `XDG_DATA_HOME`
5. `~/.local/share/clipboard-hub`

Windows 常见路径示例：

```text
%APPDATA%\Clipboard Hub\clipboard_hub.db
```

SQLite 以 WAL 模式运行。应用退出时执行 `wal_checkpoint(TRUNCATE)`，然后关闭数据库连接。

如果数据库损坏，启动时会把旧文件重命名为 `clipboard_hub.db.corrupt-时间戳`，然后重建空库。

### 3.2 数据模型

一条 `ClipItem` 代表一次复制操作。当前主要保存文本和 HTML；文件剪贴板会转换为本地路径文本。图片字段保留在模型里用于兼容旧记录，但当前捕获和写回路径不再产生新的图片记录。

```python
ClipItem(
    id: str,
    text: str | None,
    html: str | None,
    image_png: bytes | None,           # 兼容旧数据；当前新记录不再写入图片
    image_size: tuple[int, int] | None, # 兼容旧数据；当前新记录不再写入图片
    preview: str,
    timestamp: float,
    source_app: str,
    source_window: str | None,
    content_hash: str,
    size_bytes: int,
)
```

数据库核心字段包含：

- 文本内容
- HTML 内容
- PNG 二进制字段，兼容旧数据
- 图片宽高字段，兼容旧数据
- 预览文本
- 来源应用和窗口标题
- 内容哈希
- 总字节数

启动时按时间倒序载入历史。新增、删除和自动淘汰会同步写入数据库。

### 3.3 保留策略

当前同时执行两项限制：

- 最多保留 1000 条记录
- 总载荷大小最多 `200 * 1024 * 1024` 字节

超过限制时，从最旧记录开始淘汰。

额外规则：

- 单条记录如果自身已超过总容量上限，会直接丢弃，不会挤掉已有历史。
- 图片剪贴板当前不入库；如果同一剪贴板里还有文本或 HTML，只保留文本/HTML。
- 文件复制会通过 `QMimeData.urls()` 转换为换行分隔的本地路径文本。

## 四、剪贴板捕获与写回

### 4.1 捕获

Watcher 会尽量读取：

- `text/plain`
- `text/html`
- 文件 URL，并转换成本地路径文本

文本和 HTML 可属于同一个 `ClipItem`，不会拆成多条历史。图片剪贴板不再生成图片历史项；文件复制会显示为路径文本。

### 4.2 写回

重新复制历史项时，`ClipboardCodec.encode()` 只写回纯文本：

- 文本走 `setText()`
- HTML 历史项会先剥离标签，作为纯文本写回
- 图片字段即使存在于旧记录，也不会作为图片写回
- 文件路径历史项按路径文本写回

当前产品口径是“历史浏览和重新复制都以纯文本为准”，避免富文本、图片或文件对象在重新复制时带来额外格式副作用。

### 4.3 去重与自复制

- 当前只跳过“连续重复内容”，不是全局去重。
- 去重依据是当前记录文本/HTML组合后的 `content_hash`；旧图片字段仍可能参与旧记录的原始哈希。
- 从历史记录重新复制前，会先计算“实际写回纯文本”的哈希并通知 Watcher。
- Watcher 跳过自复制内容后，会同步更新 `_last_seen_hash`，避免下一轮轮询又把同一内容当作新记录加入。
- Watcher 只跳过与该哈希精确匹配的自身写回，不会无条件忽略后续用户复制。

## 五、来源应用

当前实现通过 Windows 前台窗口 API 尽力获取：

- 前台进程可执行文件名，例如 `chrome.exe`
- 前台窗口标题，例如网页标题或文档名

保存字段：

- `source_app`：获取失败时为 `unknown`
- `source_window`：窗口标题，可为空

当前实现仍会采集 `source_app` 与 `source_window`，用于搜索与后续扩展；但最新 UI 默认不展示来源标签或窗口标题，避免干扰内容浏览。

## 六、屏幕与吸附行为

- TabBar 可在多个显示器之间拖动。
- 松手 600ms 后，吸附到当前显示器右上角。
- 如果 TabBar 跨越多个显示器，以当前锚点所在显示器为准。
- 拖动过程中会同步计算面板位置。
- Panel 优先在标签下方展开；空间不足时向上展开，并尽量保持在当前屏幕范围内。

说明：当前 `HistoryPanel` 与 `TabBar` 已共享同一份 `PANEL_SIZE = (420, 520)`，首次展开和拖动后的面板定位也统一走 `screen_geometry.panel_position(...)`。

## 七、视觉设计

### 7.1 主题

当前主题模块支持三种模式解析：

- `light`
- `dark`
- `system`

主程序入口默认使用 `system` 模式；`HistoryPanel` 和 `TabBar` 都会在系统主题变化时刷新自身样式。

### 7.2 TabBar

- 默认尺寸约 `120 x 8`
- 拖动时宽度会临时增至 `132`
- 胶囊形态，贴靠屏幕右上边缘
- 当前视觉跟随面板做冷白玻璃化处理，作为面板入口条而非单纯细线悬浮条

### 7.3 HistoryPanel

- 固定尺寸 `420 x 520`
- 顶部搜索框带原生清空按钮
- 历史项采用紧凑文本复制框布局
- 空历史与无搜索结果使用不同空状态文案
- 当前 UI 已完成中文化，视觉方向为“冷白微蓝 / iOS 透明玻璃 / 紧凑文本列表”
- 图片项不再渲染图片预览；如果复制的是文件，会显示文件路径文本
- 历史项采用“倒序序号 + 最多两行内容 + 右侧等高删除覆盖条 + 柔和渐变分割线”的列表结构
- 参考图只作为布局参考，不作为颜色参考；颜色口径以当前实现的高级冷白玻璃风为准
- 每个复制框上下边距一致
- 每个复制框固定为比搜索框略短的宽度，列表不显示横向滚动条
- 复制框最多显示两行内容，超出部分通过底部渐隐遮罩淡化隐藏
- 条目分割线为更薄、更融入面板材质的横向渐变线
- 删除按钮为复制框右侧等高覆盖条，宽度为高度的二分之一，保持与复制框一致的圆角语言

### 7.4 反馈与收回

当前实现文案已统一中文：

- 复制成功：`已复制`
- 复制失败：`复制失败`
- 空历史：`还没有剪贴板记录`
- 无搜索结果：`没有找到匹配内容`

时间文案也已本地化，例如：

- `刚刚`
- `1 分钟前`
- `2 小时前`
- `3 天前`

反馈持续约 1200ms，然后恢复默认状态。

- 复制成功/失败反馈是状态徽标；点击复制后除复制框状态变色外，还会显示 `已复制` 文案。
- 元信息区默认仅保留独立时间徽标，不显示来源标签。
- 搜索输入期间视为持续交互状态，不能触发面板自动收回。
- 点击历史项完成复制后，鼠标移出面板应正常触发自动收回；复制后的内部焦点不会阻止收回。

## 八、搜索

搜索范围包括：

- 纯文本
- 原始 HTML
- 去标签后的 HTML 文本
- `preview`
- 来源应用名
- 来源窗口标题

搜索大小写不敏感。当前数据量上限为 1000 条，因此直接在内存中过滤，不引入 SQLite FTS。

## 九、错误处理

- 数据库损坏：备份损坏文件后重建空库
- 剪贴板暂时不可访问：跳过本轮，记录异常并等待下一轮
- 来源应用获取失败：保存为 `unknown`
- HTML 或文本读取失败：尽量保留本次复制中的其他文本格式
- 图片剪贴板：不保存图片载荷；如果同时存在文本或 HTML，则保留文本/HTML
- 文件剪贴板：转换为本地路径文本
- 重新复制失败：显示 `复制失败`

说明：当前主程序入口没有实现“数据库打开失败后降级为纯内存模式”，文档和 README 需要按此现状描述。

## 十、隐私边界

- 数据仅存储在本机

- 来源窗口标题会被采集并可被搜索命中，但默认不直接显示在历史卡片中

## 十四、2026-06-24 问题修复记录

本轮用户反馈的问题与最终目标行为如下：

1. 面板内点击复制后会重新生成同内容记录。
   - 目标行为：重新复制不会新增历史。
   - 当前实现：复制前通知 Watcher 实际写回纯文本的哈希；Watcher 跳过该内容后同步更新 `_last_seen_hash`，避免下一轮轮询再次入库。
2. 复制图片功能去掉，只复制纯文字；复制文件则显示文件路径。
   - 目标行为：图片不入库、不预览、不写回；文件 URL 转为换行分隔的本地路径文本。
   - 当前实现：`ClipboardCodec.decode()` 只生成文本/HTML记录，文件 URL 转路径；`ClipboardCodec.encode()` 只写 `setText()`。
3. 复制框与分割线的上下边距不一致。
   - 目标行为：复制框上下边距一致，视觉节奏稳定。
   - 当前实现：复制框外层布局上下边距对称，分割线高度收窄。
4. 复制框高度过高、内容堆积，长内容应最多两行并淡化隐藏。
   - 目标行为：复制框更矮，内容左对齐且垂直居中，最多两行，超出部分渐隐。
   - 当前实现：复制框固定高度 `52`，预览区限制两行高度并增加底部渐隐遮罩。
5. 每条复制内容添加倒序序号。
   - 目标行为：最新列表中显示倒序序号。
   - 当前实现：HistoryList 渲染时按当前可见列表数量传入倒序序号。
6. 分割线需要和面板更融为一体。
   - 目标行为：分割线更薄、更轻，符合玻璃材质。
   - 当前实现：条目分割线为 `2px` 横向渐变线。
7. 复制框和搜索框宽度关系。
   - 目标行为：复制框略短于搜索框，不出现横向滚动条。
   - 当前实现：复制框固定宽度 `360`，列表横向滚动条策略为 `AlwaysOff`。
8. 删除按钮形态。
   - 目标行为：删除按钮作为复制框最右侧等高覆盖条，宽度为高度的二分之一。
   - 当前实现：删除按钮尺寸为 `26 x 52`，与复制框等高并保持圆角。
9. 点击复制后，鼠标移出面板不能收回。
   - 目标行为：搜索输入期间保持展开；点击复制成功后，鼠标移出面板应正常收回。
   - 当前实现：复制成功后设置一次性允许收回状态，避免内部焦点阻止隐藏。

## 十五、后续改进路线

以下内容不是当前已实现范围，而是下一阶段最值得推进的产品化方向。优先级按“对日常使用价值”和“对稳定性的帮助”排序。

### 15.1 数据路径与应用身份

当前代码会显式设置 Qt application name / organization name：

- organization name：`Greasionix`
- application name：`Clipboard Hub`

因此默认数据库路径会随 `QStandardPaths.AppDataLocation` 稳定落在 `Clipboard Hub` 应用目录，而不是 Python 运行时目录。

目标价值：

- 避免用户找不到数据库位置。
- 避免路径漂移造成用户误以为历史丢失。
- 提升桌面应用的成品感。

### 15.2 设置入口与隐私开关

建议增加一个轻量设置入口，可先从托盘菜单进入。首批设置项建议包括：

- 是否记录来源应用和窗口标题
- 最大历史数量
- 主题模式：`system` / `light` / `dark`
- 是否开机启动
- 清空历史

其中“是否记录来源应用和窗口标题”优先级最高。当前 UI 默认不展示来源信息，但数据仍会采集并可被搜索命中；如果用户担心隐私，应允许完全关闭采集。

### 15.3 全局快捷键

建议增加全局快捷键打开面板，例如 `Ctrl+Shift+V`，让工具不完全依赖右上角 TabBar。

注意事项：

- 快捷键必须可配置或可关闭。
- 需要避免与系统、输入法和常见软件快捷键冲突。
- 快捷键触发展开后仍应沿用当前搜索、复制、鼠标移出收回逻辑。

### 15.4 清空历史与批量管理

当前支持逐条删除。后续建议补充：

- 托盘菜单里的“清空历史”
- 清空前确认
- 清空后立即刷新面板

更进一步可以考虑批量选择，但第一阶段只做“清空历史”即可。

### 15.5 搜索体验增强

当前搜索已覆盖文本、HTML、preview、来源应用和窗口标题。后续可增强：

- 高亮匹配词
- 搜索结果中保留原始历史序号或明确当前结果序号语义
- 搜索框聚焦时自动选中文本，方便快速替换关键词

这些改动应保持搜索输入期间面板不自动收回。

### 15.6 操作区可发现性

当前删除按钮是右侧等高覆盖条，视觉噪音低，但首次使用时可发现性仍可继续优化。后续建议：

- 给删除按钮增加非常轻的 `x` 或垃圾桶图标
- 增加 tooltip：`删除`
- 增加可访问名称，方便辅助技术识别

图标应保持克制，不破坏当前内容优先的列表风格。
